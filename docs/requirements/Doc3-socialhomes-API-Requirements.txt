1. API OVERVIEW
================================================================================

1.1 API Strategy
    SocialHomes.Ai uses a server-side proxy pattern for ALL external API
    integrations. The React frontend communicates exclusively with the Express
    backend, which handles external calls, caching, rate limiting, and
    graceful degradation.

1.2 Integration Tiers

    TIER 1 — Free, No API Key Required
    Publicly available APIs with no registration needed.
    Status: LIVE with real data

    TIER 2 — Free, Registration Required
    APIs requiring account creation or key registration.
    Status: LIVE with real data (where keys available) + simulated fallbacks

    TIER 3 — Subscription/Paid APIs
    Commercial APIs requiring contracts or paid subscriptions.
    Status: MOCK endpoints with realistic simulated data

1.3 Current Integration Count
    Total: 20 integrations
    - Tier 1: 3 live
    - Tier 2: 6 live (with simulated fallbacks)
    - Tier 3: 5 mock services + 6 planned


2. TIER 1 INTEGRATIONS (Free, No Key)
================================================================================

2.1 Police.uk — Street-Level Crime Data
    Status: LIVE
    Base URL: https://data.police.uk/api
    Documentation: https://data.police.uk/docs/

    Endpoints Used:
    - GET /crimes-street/all-crime?lat={lat}&lng={lng}&date={YYYY-MM}
    - GET /crimes-at-location?location_id={id}
    - GET /outcomes-at-location?location_id={id}

    Server Route: GET /api/v1/public-data/police/:postcode
    Cache TTL: 24 hours
    Rate Limit: 15 req/sec (fair-use)
    Fallback: Simulated crime data matching postcode area

    Used By:
    - D2: Live Crime Context for ASB (crime-context.ts)
    - Explore page crime data layer
    - ASB case correlation
    - Neighbourhood Briefing crime section

    Data Returned:
    - Crime category (anti-social-behaviour, burglary, criminal-damage, etc.)
    - Month, location (lat/lng), outcome status
    - 12-month historical data per location

2.2 Open-Meteo — Weather Forecasting
    Status: LIVE
    Base URL: https://api.open-meteo.com/v1
    Documentation: https://open-meteo.com/en/docs

    Endpoints Used:
    - GET /forecast?latitude={lat}&longitude={lng}&hourly=temperature_2m,
          relative_humidity_2m,precipitation,weathercode,windspeed_10m
          &daily=temperature_2m_max,temperature_2m_min,precipitation_sum
          &timezone=Europe/London

    Server Route: GET /api/v1/public-data/weather/:lat/:lng
    Cache TTL: 1 hour
    Rate Limit: 10,000 req/day (no key needed)
    Fallback: Static weather for London SE

    Used By:
    - D1: Predictive Damp Intelligence (weather factor 0.25 weight)
    - AI Morning Briefing (weather section)
    - Neighbourhood Briefing (environmental context)
    - Property impact assessment

    Data Returned:
    - Hourly: temperature, humidity, precipitation, wind speed, weather code
    - Daily: max/min temperature, precipitation sum
    - 7-day forecast + current conditions

5. SEVEN KILLER DIFFERENTIATOR SERVICES
================================================================================

5.1 D1: Predictive Damp Intelligence
    Service: server/src/services/damp-prediction.ts
    API Route: GET /api/v1/ai/damp-risk/:propertyId

    Algorithm: 5-Factor Weighted Score (0–100)
    ┌─────────────────────┬────────┬────────────────────────────────┐
    │ Factor              │ Weight │ Data Sources                   │
    ├─────────────────────┼────────┼────────────────────────────────┤
    │ Weather conditions  │  0.25  │ Open-Meteo (humidity, rain)    │
    │ Building attributes │  0.25  │ Property data (age, type, EPC) │
    │ Repair history      │  0.20  │ Case data (damp/mould reports) │
    │ IoT sensors         │  0.20  │ Switchee/Aico (temp, humidity) │
    │ Occupancy patterns  │  0.10  │ Tenant data (household size)   │
    └─────────────────────┴────────┴────────────────────────────────┘

    Output:
    - Overall risk score (0–100)
    - Risk level: low (0–25), moderate (26–50), high (51–75), critical (76–100)
    - Factor breakdown with individual scores
    - Proactive recommendations
    - Awaab's Law compliance status

5.2 D2: Live Crime Context for ASB
    Service: server/src/services/crime-context.ts
    API Route: GET /api/v1/ai/crime-context/:estateId

    Algorithm:
    - Fetch Police.uk data for estate location (12-month window)
    - Categorize by type (ASB, violence, drugs, criminal damage, etc.)
    - Correlate with internal ASB cases
    - Identify hotspots and trends
    - Generate estate safety score

    Output:
    - Crime summary by category (last 12 months)
    - Monthly trend (rising/stable/declining)
    - Hotspot locations within estate
    - Correlation with internal ASB cases
    - Recommended patrol priorities

5.3 D3: Automatic Vulnerability Detection
    Service: server/src/services/vulnerability-detection.ts
    API Route: GET /api/v1/ai/vulnerability/:tenantId

    Algorithm: 7-Factor Weighted Score (0–100)
    ┌──────────────────────────┬────────┬──────────────────────────┐
    │ Factor                   │ Weight │ Data Sources             │
    ├──────────────────────────┼────────┼──────────────────────────┤
    │ IMD deprivation          │  0.20  │ IMD data (decile)        │
    │ Arrears history          │  0.20  │ Rent account (balance)   │

    │ Social isolation         │  0.15  │ Household composition    │
    │ Age risk (elderly/young) │  0.10  │ Tenant DOB              │
    │ Dependents               │  0.10  │ Household members        │
    │ UC transition            │  0.10  │ UC status                │

7. CACHING ARCHITECTURE
================================================================================

7.1 Cache Collection: externalDataCache
    Stored in Firestore with TTL-based expiration.
    Cache key format: "{source}:{params}" (e.g., "police:SE17+1RY")

7.2 Cache-Through Pattern
    1. Check Firestore cache for key
    2. If found AND not expired → return cached data
    3. If missing OR expired → fetch from external API
    4. Store result in cache with TTL
    5. Return data to caller

7.3 TTL Configuration

    | Source              | TTL       | Key Format                 |
    |---------------------|-----------|----------------------------|
    | police              | 24 hours  | police:{postcode}          |
    | weather             | 1 hour    | weather:{lat},{lng}        |
    | flood-risk          | 1 hour    | flood:{postcode}           |
    | postcode            | 7 days    | postcode:{postcode}        |
    | epc                 | 30 days   | epc:{uprn}                 |
    | census              | 90 days   | census:{lsoaCode}          |
    | nomis               | 30 days   | nomis:{lsoaCode}           |
    | gas-safe            | 24 hours  | gas-safe:{regNumber}       |
    | electrical-safety   | 24 hours  | electrical:{regNumber}     |
    | imd                 | 90 days   | imd:{lsoaCode}             |

7.4 Cache Management
    - Automatic TTL enforcement on read
    - Manual cache invalidation via admin API
    - Cache hit rate tracked in metrics
    - No cache warming (fetch-on-demand only)

9. ERROR HANDLING & GRACEFUL DEGRADATION
================================================================================

9.1 External API Error Handling
    - Timeout: 10 seconds per external request
    - Retry: 1 retry with exponential backoff (2s)
    - On failure: Return simulated fallback data
    - Flag: Response includes { source: 'live' | 'simulated' }

9.2 Simulated Fallback Strategy
    Every external API has a corresponding simulated data generator.
    Simulated data is:
    - Realistic for the requested entity/location
    - Consistent across requests (deterministic seeding)
    - Flagged clearly in the response

9.3 Circuit Breaker Pattern (Planned)
    - Track failure count per API source
    - Open circuit after 5 consecutive failures
    - Half-open after 60 seconds (single test request)
    - Close circuit on successful test
    Status: Not yet implemented (deferred)


10. API SECURITY
================================================================================

10.1 Authentication
    All /api/v1/* routes require Firebase Bearer token.
    Exception: /health (public health check)

10.2 Request Validation
    - JSON body limit: 1MB
    - Content-Type: application/json enforced
    - Query parameter sanitization
    - Path parameter validation

10.3 Response Security
    - No sensitive data in error responses (production)
    - CORS restricted to allowed origins
    - Request ID correlation (X-Request-ID)
    - Security headers on all responses

10.4 API Key Management
    - No API keys in source code
    - All keys in Google Secret Manager
    - Environment variable injection at runtime
    - Key rotation support via Secret Manager versioning
