# Repair Workflow Overhaul — AI-Native, Professional, Complete

## Context

The repair detail page is a 547-line monolith with a flat layout and disconnected actions. User feedback:
- Workflow is confusing, "going all over the place"
- No labour or parts cost tracking
- No way to properly close a repair (Close button sets `completed`, not `closed`)
- No linked repairs, linked complaints, or duplicate functionality
- No AI intelligence in the workflow lifecycle — only at creation
- "Handler" is unprofessional — should be "Assigned Officer"
- Should show tenant vulnerability flags and property damp/mould history

## Approach — Single-file rewrite + targeted supporting changes

Rather than decomposing into 9 sub-components (over-engineering for current stage), keep `RepairDetailPage.tsx` as one file with clear internal sections, add a tabbed layout, and make targeted changes to 6 supporting files.

### Tabs

**Details Tab** — Case description, trade, operative, dates, cost summary, recurrence risk. Extracted from current lines 381-473. Rename "Handler" → "Assigned Officer".

**Costs Tab** — NEW
- Cost summary card at top: Labour £X + Parts £X = Total £X (using `formatCurrency`)
- Labour log table: Date | Operative | Hours | Rate | Total | Description
- Parts log table: Date | Part | Qty | Unit Cost | Total | Supplier
- "Add Labour" and "Add Parts" buttons open ActionModal with number fields
- On submit: append to `repair.labourLog[]` / `repair.partsLog[]` via `casesApi.update()`
- Firestore arrays need full replacement — read existing, append, send complete array

**Linked Cases Tab** — NEW
- List of linked repairs and complaints with: Reference (link) | Type badge | Subject | Status pill | Date
- Fetched by filtering `repairs`/`complaints` where ID is in `repair.linkedCases` or `repair.linkedComplaints`
- Three action buttons:
  - **Create Linked Repair**: `casesApi.create()` with same property/tenant, `parentCaseId: repair.id`, then update current repair's `linkedCases[]`
  - **Raise Complaint**: `casesApi.create()` with `type: 'complaint'`, same property/tenant, `category: 'Repairs & Maintenance'`, then update current repair's `linkedComplaints[]`
  - **Duplicate**: `casesApi.create()` copying key fields (subject, description, sorCode, trade, priority, propertyId, tenantId) with fresh dates/status, `duplicatedFrom: repair.id`

**Activity Tab** — Activity feed. Extracted from current lines 492-516. No changes to content.

## 6. Activity Logging on Status Changes

**File:** `server/src/routes/cases.ts`

In the PATCH endpoint, after `updateDoc`, if `req.body.status` is present and different from current, create an activity:
```typescript
if (req.body.status) {
  await collections.activities.add({
    caseId: req.params.id,
    tenantId: updated.tenantId,
    type: 'system',
    subject: `Status changed to ${req.body.status}`,
    description: `Case ${updated.reference} status updated from ${oldStatus} to ${req.body.status}`,
    date: new Date().toISOString().split('T')[0],
    officer: req.user?.displayName || 'System',
  });
}
```
